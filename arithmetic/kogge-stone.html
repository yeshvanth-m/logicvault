Parallel Prefix Computation

#

The Kogge-Stone adder is based on parallel prefix computation. Like all prefix adders, it computes Generate (G) and Propagate (P) signals, then combines them using a prefix operator to compute all carries in parallel.

Initial P and G

For each bit position i, compute the initial generate and propagate signals from the input operands.

Gi = Ai · Bi
Pi = Ai ⊕ Bi
Prefix Operator (•)

The associative prefix operator combines (G,P) pairs from different bit ranges to produce group generate and propagate.

(Gi:k, Pi:k) = (Gi:j, Pi:j) • (Gj-1:k, Pj-1:k)

Group Generate

Gi:k = Gi:j + (Pi:j · Gj-1:k)

Group Propagate

Pi:k = Pi:j · Pj-1:k

Final Carry and Sum

Ci = Gi-1:0 + (Pi-1:0 · C0)

Sumi = Pi ⊕ Ci

Key insight: The prefix operator is associative, allowing parallel computation. Kogge-Stone maximizes parallelism by computing all prefix sums at every level, trading area for speed.

Structure

#

Kogge-Stone uses a dense prefix tree with maximum parallelism. At each level k, every position i computes a prefix spanning 2k bits by combining adjacent groups.

Level Span Cells at level Description
01NInitial P, G computation (PG cells)
12N-1Combine adjacent pairs
24N-2Combine groups of 4
38N-4Combine groups of 8
⋮⋮⋮⋮
log₂NN1Final prefix complete

Wiring complexity: The dense structure requires O(N log N) wires with long wire spans. This can cause routing congestion and increased power consumption in physical implementations.

RTL Implementation

#
Verilog kogge_stone_8bit.v 
// 8-bit Kogge-Stone Parallel Prefix Adder
module kogge_stone_8bit (
    input  [7:0] a, b,
    input        cin,
    output [7:0] sum,
    output       cout
);

    wire [7:0] p0, g0;         // Level 0: initial P, G
    wire [7:0] p1, g1;         // Level 1: span 2
    wire [7:0] p2, g2;         // Level 2: span 4
    wire [7:0] p3, g3;         // Level 3: span 8 (final)
    wire [8:0] c;              // Carries

    // Level 0: Generate initial P and G
    assign p0 = a ^ b;
    assign g0 = a & b;

    // Level 1: Combine adjacent pairs (span 2)
    assign g1[0] = g0[0];
    assign p1[0] = p0[0];
    genvar i;
    generate
        for (i = 1; i < 8; i = i + 1) begin : L1
            assign g1[i] = g0[i] | (p0[i] & g0[i-1]);
            assign p1[i] = p0[i] & p0[i-1];
        end
    endgenerate

    // Level 2: Combine span-4 groups
    assign {g2[1:0], p2[1:0]} = {g1[1:0], p1[1:0]};
    generate
        for (i = 2; i < 8; i = i + 1) begin : L2
            assign g2[i] = g1[i] | (p1[i] & g1[i-2]);
            assign p2[i] = p1[i] & p1[i-2];
        end
    endgenerate

    // Level 3: Combine span-8 groups (final)
    assign {g3[3:0], p3[3:0]} = {g2[3:0], p2[3:0]};
    generate
        for (i = 4; i < 8; i = i + 1) begin : L3
            assign g3[i] = g2[i] | (p2[i] & g2[i-4]);
            assign p3[i] = p2[i] & p2[i-4];
        end
    endgenerate

    // Final carry computation
    assign c[0] = cin;
    generate
        for (i = 0; i < 8; i = i + 1) begin : CARRY
            assign c[i+1] = g3[i] | (p3[i] & c[0]);
        end
    endgenerate

    // Sum computation
    assign sum = p0 ^ c[7:0];
    assign cout = c[8];

endmodule
VHDL kogge_stone_8bit.vhd 
-- 8-bit Kogge-Stone Parallel Prefix Adder
library IEEE;
use IEEE.STD_LOGIC_1164.all;

entity kogge_stone_8bit is
    port (
        a, b : in  std_logic_vector(7 downto 0);
        cin   : in  std_logic;
        sum   : out std_logic_vector(7 downto 0);
        cout  : out std_logic
    );
end kogge_stone_8bit;

architecture rtl of kogge_stone_8bit is
    signal p0, g0, p1, g1, p2, g2, p3, g3 : std_logic_vector(7 downto 0);
    signal c : std_logic_vector(8 downto 0);
begin
    -- Level 0: Initial P and G
    p0 <= a xor b;
    g0 <= a and b;

    -- Level 1: Span 2
    g1(0) <= g0(0);
    p1(0) <= p0(0);
    L1: for i in 1 to 7 generate
        g1(i) <= g0(i) or (p0(i) and g0(i-1));
        p1(i) <= p0(i) and p0(i-1);
    end generate;

    -- Level 2: Span 4
    g2(1 downto 0) <= g1(1 downto 0);
    p2(1 downto 0) <= p1(1 downto 0);
    L2: for i in 2 to 7 generate
        g2(i) <= g1(i) or (p1(i) and g1(i-2));
        p2(i) <= p1(i) and p1(i-2);
    end generate;

    -- Level 3: Span 8 (final)
    g3(3 downto 0) <= g2(3 downto 0);
    p3(3 downto 0) <= p2(3 downto 0);
    L3: for i in 4 to 7 generate
        g3(i) <= g2(i) or (p2(i) and g2(i-4));
        p3(i) <= p2(i) and p2(i-4);
    end generate;

    -- Final carries and sum
    c(0) <= cin;
    CARRY: for i in 0 to 7 generate
        c(i+1) <= g3(i) or (p3(i) and c(0));
    end generate;

    sum <= p0 xor c(7 downto 0);
    cout <= c(8);
end rtl;

Power consideration: Kogge-Stone's high switching activity and dense wiring can result in significant dynamic power. For power-constrained designs, consider Brent-Kung or hybrid architectures.

Interview Notes

#
  • Why is Kogge-Stone the fastest? — It achieves minimum logic depth (log₂N) by computing all prefix sums at every level, maximizing parallelism at the cost of area.
  • What's the main disadvantage? — Dense wiring (O(N log N) wires) leads to routing congestion, increased capacitance, and higher power consumption.
  • How does it compare to Brent-Kung? — Brent-Kung trades speed (2× depth) for area efficiency (O(N) wires vs O(N log N)).
  • What is the prefix operator? — An associative operation combining (G,P) pairs: Gout = Ghigh + Phigh·Glow, Pout = Phigh·Plow.
  • Where is Kogge-Stone used? — High-performance CPUs, FPUs, and DSPs where latency is critical and area/power can be traded.
  • Can you pipeline a Kogge-Stone adder? — Yes, registers can be inserted between prefix levels for higher throughput at the cost of latency.
