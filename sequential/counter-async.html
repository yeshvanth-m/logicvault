Sequential Counter

Asynchronous Counter (Ripple Counter)

An asynchronous counter chains flip-flops where the output of one stage clocks the next. Also called a ripple counter because the clock signal "ripples" through the stages with cumulative delay.

Building block

T or JK Flip-Flops

N-bit counter

Counts 0 to 2N-1

Max frequency

fmax = 1 / (N × tpd)

Complexity

Simple (minimal logic)

How It Works

#

Each flip-flop is configured in toggle mode (T=1 or JK with J=K=1). The output Q of each stage clocks the next stage:

  • Stage 0: Clocked by the input clock
  • Stage 1: Clocked by Q0 (toggles when Q0 falls/rises)
  • Stage N: Clocked by Q(N-1)

Timing & Propagation Delay

#

The key limitation: each stage adds one flip-flop delay. For an N-bit counter, total worst-case delay is N × tpd.

Glitch hazard: During the ripple period, outputs show intermediate (incorrect) values. Example: 0111→1000 may briefly show 0110, 0100, 0000 before settling to 1000.

Up Counter

#

Counts upward: 0, 1, 2, ... 2N-1, then wraps to 0. Uses negative edge triggering — each stage toggles when the previous stage transitions from 1→0.

Count Q2Q1Q0
0000
1001
2010
3011
4100
5101
6110
7111

Down Counter

#

Counts downward: 2N-1, ... 2, 1, 0, then wraps to 2N-1. Uses Q' (complement) output to clock next stage, or uses positive edge triggering.

Method 1

Use Q' to clock next

Same negative-edge FFs, but connect Q' instead of Q

Method 2

Use positive-edge FFs

Toggle on 0→1 transition of previous stage

MOD-N Counter (Truncated Counter)

#

Counts from 0 to N-1 (not a power of 2). Uses asynchronous reset to force counter back to 0 when target count is reached.

MOD-10 (Decade Counter)

0 → 9 → 0

Reset when Q3Q2Q1Q0 = 1010 (10)

MOD-6

0 → 5 → 0

Reset when Q2Q1Q0 = 110 (6)

Glitch in reset: When using combinational logic to detect count N and trigger async reset, the counter momentarily shows N before resetting. This can cause glitches in downstream logic.

RTL (Verilog / VHDL)

#
Verilog ripple_counter.v 
// 4-bit Ripple Counter (Asynchronous)
// Note: This models the ripple behavior but synthesis
// tools may optimize it differently
module ripple_counter_4bit (
    input  wire clk,
    input  wire rst_n,
    output reg [3:0] Q
);

    // Stage 0 - clocked by input clock
    always @(negedge clk or negedge rst_n) begin
        if (!rst_n)
            Q[0] <= 1'b0;
        else
            Q[0] <= ~Q[0];  // Toggle
    end

    // Stage 1 - clocked by Q[0]
    always @(negedge Q[0] or negedge rst_n) begin
        if (!rst_n)
            Q[1] <= 1'b0;
        else
            Q[1] <= ~Q[1];
    end

    // Stage 2 - clocked by Q[1]
    always @(negedge Q[1] or negedge rst_n) begin
        if (!rst_n)
            Q[2] <= 1'b0;
        else
            Q[2] <= ~Q[2];
    end

    // Stage 3 - clocked by Q[2]
    always @(negedge Q[2] or negedge rst_n) begin
        if (!rst_n)
            Q[3] <= 1'b0;
        else
            Q[3] <= ~Q[3];
    end

endmodule

// MOD-10 Decade Counter with Async Reset
module decade_counter (
    input  wire clk,
    input  wire rst_n,
    output reg [3:0] Q
);

    wire reset_10 = (Q == 4'd10);  // Decode count 10

    always @(posedge clk or negedge rst_n or posedge reset_10) begin
        if (!rst_n || reset_10)
            Q <= 4'b0;
        else
            Q <= Q + 1'b1;
    end

endmodule
VHDL ripple_counter.vhd 
-- 4-bit Ripple Counter (Asynchronous)
library IEEE;
use IEEE.STD_LOGIC_1164.all;

entity ripple_counter_4bit is
    port (
        clk   : in  std_logic;
        rst_n : in  std_logic;
        Q     : out std_logic_vector(3 downto 0)
    );
end ripple_counter_4bit;

architecture behavioral of ripple_counter_4bit is
    signal Q_int : std_logic_vector(3 downto 0);
begin
    -- Stage 0
    process(clk, rst_n)
    begin
        if rst_n = '0' then
            Q_int(0) <= '0';
        elsif falling_edge(clk) then
            Q_int(0) <= not Q_int(0);
        end if;
    end process;

    -- Stage 1
    process(Q_int(0), rst_n)
    begin
        if rst_n = '0' then
            Q_int(1) <= '0';
        elsif falling_edge(Q_int(0)) then
            Q_int(1) <= not Q_int(1);
        end if;
    end process;

    -- Stage 2
    process(Q_int(1), rst_n)
    begin
        if rst_n = '0' then
            Q_int(2) <= '0';
        elsif falling_edge(Q_int(1)) then
            Q_int(2) <= not Q_int(2);
        end if;
    end process;

    -- Stage 3
    process(Q_int(2), rst_n)
    begin
        if rst_n = '0' then
            Q_int(3) <= '0';
        elsif falling_edge(Q_int(2)) then
            Q_int(3) <= not Q_int(3);
        end if;
    end process;

    Q <= Q_int;
end behavioral;

Limitations & When to Use Sync Counters

#
Issue Async Counter Sync Counter
Propagation delay N × tpd (cumulative) 1 × tpd (all parallel)
Max frequency Limited by ripple time Higher (clock period only)
Glitches Yes (transient states) No (all bits change together)
Logic complexity Minimal More (carry logic)
Power Lower Higher

When to use async: Low-speed applications, simple frequency dividers, low-power designs. For anything feeding synchronous logic or requiring glitch-free outputs, use synchronous counters.
