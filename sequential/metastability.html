Metastability

When a flip-flop samples an input during its setup/hold window, the output can enter an unstable intermediate state. Understanding metastability is essential for reliable clock domain crossing design.

Cause

Setup/hold violation

Result

Undefined output

Solution

Synchronizers

Metric

MTBF

What Causes Metastability

#

Every flip-flop has a setup time (tsu) and hold time (th) — a window around the clock edge where the input must be stable. If the input transitions during this window:

The Physics

The FF's internal latch is a bistable circuit. During violation, neither stable state "wins" — the circuit hangs at an intermediate voltage between 0 and 1.

The Problem

This intermediate voltage may be interpreted as 0 by some gates and 1 by others, causing logic corruption and unpredictable behavior.

Resolution Time

#

The metastable state doesn't last forever — thermal noise eventually pushes the output to a valid logic level. But how long this takes is probabilistic.

Probability of remaining metastable after time t

P(t) = e-t/τ

where τ is the resolution time constant (technology dependent, typically 20-100ps)

Key insight: We can reduce the probability exponentially by giving more time to resolve. This is why multi-stage synchronizers work — each stage provides more resolution time.

MTBF Calculation

#

Mean Time Between Failures (MTBF) quantifies reliability. Higher is better — we typically want MTBF > 100 years for production designs.

MTBF Formula

MTBF = etr/τ / (T0 · fclk · fdata)

Parameter Description Typical Value
trResolution time (slack for synchronizer)Clock period - tco - tsu
τResolution time constant20-100 ps
T0Metastability window~setup + hold time
fclkSampling clock frequencyDesign dependent
fdataData transition frequencyDesign dependent

Example: At 500MHz with τ=50ps, a 2-FF synchronizer with 1ns resolution time gives MTBF ≈ 108 years. Adding a 3rd FF can increase this to 1015 years!

Synchronizer Design

#

The standard solution is a 2-FF synchronizer — two flip-flops in series, both clocked by the destination domain.

FF1 (Metastable)

May go metastable when sampling async input. Has one full clock cycle to resolve before FF2 samples.

FF2 (Stable)

By the time FF2 samples, FF1 has (probably) resolved. FF2 output is safe to use in destination logic.

Why 2 FFs? One FF gives time tr to resolve. Two FFs give 2×tr, reducing failure probability exponentially. Some designs use 3 FFs for extreme reliability.

RTL (Verilog / VHDL)

#
Verilog synchronizer.v 
// 2-FF Synchronizer
module sync_2ff (
    input  wire clk,        // Destination clock
    input  wire rst_n,
    input  wire async_in,   // Asynchronous input
    output wire sync_out    // Synchronized output
);

    // Synchronizer chain - ASYNC_REG attribute for FPGA
    (* ASYNC_REG = "TRUE" *) reg [1:0] sync_reg;

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n)
            sync_reg <= 2'b0;
        else
            sync_reg <= {sync_reg[0], async_in};
    end

    assign sync_out = sync_reg[1];

endmodule

// 3-FF Synchronizer (higher MTBF)
module sync_3ff (
    input  wire clk,
    input  wire rst_n,
    input  wire async_in,
    output wire sync_out
);

    (* ASYNC_REG = "TRUE" *) reg [2:0] sync_reg;

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n)
            sync_reg <= 3'b0;
        else
            sync_reg <= {sync_reg[1:0], async_in};
    end

    assign sync_out = sync_reg[2];

endmodule

// Synchronizer with edge detection
module sync_pulse (
    input  wire clk,
    input  wire rst_n,
    input  wire async_in,
    output wire sync_pulse  // One-cycle pulse on rising edge
);

    reg [2:0] sync_reg;  // 2 for sync + 1 for edge detect

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n)
            sync_reg <= 3'b0;
        else
            sync_reg <= {sync_reg[1:0], async_in};
    end

    // Rising edge: was 0, now 1
    assign sync_pulse = sync_reg[1] & ~sync_reg[2];

endmodule
VHDL synchronizer.vhd 
-- 2-FF Synchronizer
library IEEE;
use IEEE.STD_LOGIC_1164.all;

entity sync_2ff is
    port (
        clk       : in  std_logic;
        rst_n     : in  std_logic;
        async_in  : in  std_logic;
        sync_out  : out std_logic
    );
end sync_2ff;

architecture rtl of sync_2ff is
    -- ASYNC_REG attribute for Xilinx
    signal sync_reg : std_logic_vector(1 downto 0);
    attribute ASYNC_REG : string;
    attribute ASYNC_REG of sync_reg : signal is "TRUE";
begin

    process(clk, rst_n)
    begin
        if rst_n = '0' then
            sync_reg <= (others => '0');
        elsif rising_edge(clk) then
            sync_reg <= sync_reg(0) & async_in;
        end if;
    end process;

    sync_out <= sync_reg(1);

end rtl;

Interview Notes

#
  • What is metastability? — A state where FF output is at an intermediate voltage (neither 0 nor 1) due to setup/hold violation. It resolves probabilistically over time.
  • Can metastability be eliminated? — No, but its probability can be reduced to negligible levels (MTBF > system lifetime) using synchronizers.
  • Why 2 FFs, not 1? — One FF can go metastable. The second FF gives the first a full clock cycle to resolve before its output is sampled.
  • When would you use 3 FFs? — For very high-frequency designs or safety-critical systems where even higher MTBF is needed.
  • What's the ASYNC_REG attribute? — Tells synthesis tools to place synchronizer FFs close together, minimizing routing delay and maximizing resolution time.
  • Can you synchronize a bus? — Not safely with just 2-FF sync! Use gray coding, handshake protocols, or async FIFOs for multi-bit data.
