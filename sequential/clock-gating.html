Sequential Power

Clock Gating

The most effective technique for reducing dynamic power in synchronous designs. Clock gating stops the clock to idle flip-flops, eliminating unnecessary switching activity.

Power Savings

30-70%

Implementation

ICG Cell

Timing Critical

Enable setup

Tool Support

Automatic

Why Clock Gating?

#

In CMOS circuits, dynamic power dominates: P = α × C × V² × f. The clock network consumes 30-50% of total dynamic power because:

Clock Network Issues
  • Toggles every cycle (α = 1)
  • High capacitance (long wires, many FFs)
  • Drives every flip-flop
  • Cannot stop for idle modules
Clock Gating Benefits
  • Stops clock to idle FFs
  • Reduces switching power
  • Eliminates unnecessary toggles
  • Can gate entire subsystems

Dynamic Power Equation

Pdynamic = α × C × V² × f

α = switching activity, C = load capacitance, V = voltage, f = frequency

Integrated Clock Gating (ICG) Cell

#

The standard approach uses a latch-based ICG cell. The latch captures the enable during the low phase, producing a glitch-free gated clock.

Why a latch? The latch is transparent when clk=0, capturing enable. When clk goes high, the latch holds the enable value. The AND gate then produces a clean gated clock with no glitches.

Timing Considerations

#

Clock gating introduces new timing paths that must be analyzed:

Enable Setup Time

Enable must be stable before the falling edge of clock (when latch becomes transparent). This is a setup constraint to the ICG.

Clock Skew

Gated clock adds ICG delay. The clock tree must balance this to maintain skew requirements between gated and ungated FFs.

Timing Check Description
Enable SetupEnable stable before clk↓ (to latch)
Enable HoldEnable stable after clk↓
Clock-to-GclkICG propagation delay
Min Pulse WidthGated clock pulse must meet FF requirements

Glitch hazard: If enable changes during clk=1, the AND gate can produce glitches on gclk. The latch prevents this by holding the enable value stable during clk=1.

Clock Gating Patterns

#
RTL-Inferred Gating

Synthesis tools automatically insert ICG cells when they detect conditional register updates:

if (enable)
    q <= d;  // Tool inserts ICG
Module-Level Gating

Gate the clock to entire modules when idle:

// Explicit ICG for module
icg u_icg (
  .clk(clk),
  .en(module_active),
  .gclk(gclk_module)
);

RTL (Verilog / VHDL)

#
Verilog clock_gating.v 
// Latch-based ICG Cell (behavioral model)
module icg_cell (
    input  wire clk,
    input  wire enable,
    output wire gclk
);

    reg enable_latch;

    // Latch transparent when clk = 0 (low-active latch)
    always @(*) begin
        if (!clk)
            enable_latch <= enable;
    end

    // AND gate produces gated clock
    assign gclk = clk & enable_latch;

endmodule

// RTL style that synthesis can auto-gate
module auto_gated_reg (
    input  wire       clk,
    input  wire       rst_n,
    input  wire       enable,
    input  wire [7:0] data_in,
    output reg  [7:0] data_out
);

    // Synthesis tools will insert ICG for this pattern
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n)
            data_out <= 8'b0;
        else if (enable)    // ← This condition triggers ICG insertion
            data_out <= data_in;
        // Implicit: else data_out holds
    end

endmodule

// Explicit ICG instantiation for module-level gating
module gated_subsystem (
    input  wire       clk,
    input  wire       rst_n,
    input  wire       subsys_enable,
    input  wire [7:0] data_in,
    output reg  [7:0] data_out
);

    wire gclk;

    // Explicit ICG for the entire subsystem
    icg_cell u_icg (
        .clk    (clk),
        .enable (subsys_enable),
        .gclk   (gclk)
    );

    // All FFs in subsystem use gated clock
    reg [7:0] stage1, stage2;

    always @(posedge gclk or negedge rst_n) begin
        if (!rst_n) begin
            stage1   <= 8'b0;
            stage2   <= 8'b0;
            data_out <= 8'b0;
        end else begin
            stage1   <= data_in;
            stage2   <= stage1;
            data_out <= stage2;
        end
    end

endmodule

// Test enable with scan (for DFT)
module icg_with_scan (
    input  wire clk,
    input  wire enable,
    input  wire test_mode,    // Force clock on during scan
    output wire gclk
);

    wire effective_enable = enable | test_mode;

    icg_cell u_icg (
        .clk    (clk),
        .enable (effective_enable),
        .gclk   (gclk)
    );

endmodule
VHDL icg_cell.vhd 
-- Latch-based ICG Cell
library IEEE;
use IEEE.STD_LOGIC_1164.all;

entity icg_cell is
    port (
        clk    : in  std_logic;
        enable : in  std_logic;
        gclk   : out std_logic
    );
end icg_cell;

architecture rtl of icg_cell is
    signal enable_latch : std_logic;
begin

    -- Low-active latch (transparent when clk = '0')
    process(clk, enable)
    begin
        if clk = '0' then
            enable_latch <= enable;
        end if;
    end process;

    -- Gated clock output
    gclk <= clk and enable_latch;

end rtl;

Interview Notes

#
  • Why use a latch in ICG? — The latch holds the enable stable during clk=1, preventing glitches on the gated clock when enable changes.
  • Why not just AND the clock with enable? — If enable changes while clk=1, you get glitches. The latch ensures enable is only sampled during clk=0.
  • What's the power benefit? — Typically 30-70% dynamic power reduction by stopping clock to idle FFs. Eliminates clock tree switching for those FFs.
  • What about static power? — Clock gating doesn't reduce leakage. For that, you need power gating (cutting VDD) or multi-threshold cells.
  • How does synthesis auto-insert ICG? — Tools recognize if(enable) patterns and replace the mux-style feedback with an ICG cell, saving area and power.
  • What about scan/test mode? — ICG cells need a test_mode OR with enable to force clocks on during scan testing.
  • When shouldn't you use clock gating? — For FFs that update every cycle (no power benefit), or when gating overhead exceeds savings for small groups.
