Espresso Algorithm

Espresso is a heuristic logic minimization algorithm developed at UC Berkeley. Unlike exact methods (QMC), it trades guaranteed optimality for dramatically faster execution on large functions, making it the de facto standard in EDA tools.

Developed at

UC Berkeley (1984)

Variables

Practical for 100+

Complexity

Polynomial (heuristic)

Result

Near-optimal (usually optimal)

The Algorithm

#

Espresso operates on cubes (product terms) through three main phases, iterating until no improvement is found:

Espresso Main Loop
  1. EXPAND — Make each cube as large as possible
  2. IRREDUNDANT — Remove redundant cubes
  3. REDUCE — Shrink cubes to enable better expansion
  4. Repeat until cover stabilizes (no cost reduction)
EXPAND Phase

Each cube is expanded by adding "don't cares" (-) to as many variables as possible while staying within the ON-set ∪ DC-set. Expansion priority is given to variables that appear in many cubes (improves later redundancy removal).

IRREDUNDANT Phase

Remove cubes that are completely covered by other cubes. Classifies cubes as: essential (must keep), redundant (can remove), or partially redundant (choose minimum cover).

REDUCE Phase

Shrink each cube to its smallest form that still covers its uniquely-covered minterms. This creates opportunities for the next EXPAND iteration to find better groupings.

Key insight: REDUCE + EXPAND allows escaping local minima. Unlike greedy algorithms, Espresso can "undo" decisions to find better solutions.

How Espresso Works Internally

#

Espresso uses positional cube notation where each variable has two bits:

VariableEncodingMeaning
A = 010A' in term
A = 101A in term
A = -11A eliminated
A = ∅00Empty (contradiction)
Core Principle: Cube Operations
  • Expansion: Change 01 or 10 → 11 (add don't care)
  • Reduction: Change 11 → 01 or 10 (add literal)
  • Intersection: Bitwise AND of two cubes
  • Containment: Cube A contains B if A ∩ B = B

Simple Example: EXPAND Operation

Consider cube A'BC with ON-set allowing expansion:

1 Original Cube

CubeABC
A'BC100101

Covers minterm: m3 (011)

2 Try Expand A

CubeABC
-BC110101

Now covers: m3 (011), m7 (111)

✓ Valid if m7 ∈ ON-set ∪ DC-set

3 Continue Expanding

Try expanding B (change 01 → 11):

CubeABCCovers
-BC110101m3, m7
--C111101m1, m3, m5, m7

✓ Valid if all new minterms (m1, m5) ∈ ON-set ∪ DC-set

Result

Original cube A'BC expanded to just C — eliminating 2 variables!

A'BC → C

Key Insight: Each 11 (don't care) doubles the minterms covered. A cube with n don't cares covers 2ⁿ minterms. Espresso tries to maximize don't cares while staying within allowed minterms.

Try reducing cubes to enable better expansion:

Reduce A'BC' → check if smaller cube allows merging:

  • A'BC' could reduce to A'B (if C doesn't matter) — but m0 = A'B'C' is not in ON-set
